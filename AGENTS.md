# AGENTS.md — Project Continuation Guide

> **Purpose**: This file gives a new agent session everything it needs to continue this project. Read this entire file first, then `docs/` for protocol details and `research/` for technical analysis.

## What This Project Does

Make a **Steam Deck** present itself as a **Steam Controller 2026 (SC2)** over **Bluetooth Low Energy**, so that Steam Client on a host PC recognizes it as an SC2 with full Steam Input support (trackpads, gyro, haptics, back buttons).

## Architecture

The Steam Deck runs a raw L2CAP ATT server on CID 4 bound to static random address `C2:12:34:56:78:9A`. BlueZ is still used for SMP pairing (kernel, CID 6), LE advertising, and Agent registration — but the entire GATT service layer is bypassed. The ATT server serves 87 attributes across 6 services (GAP, GATT, HID, Battery, Device Info, Valve Custom SC2).

`input_handler.py` reads 64-byte HID reports from the Neptune controller at `/dev/hidraw3` (USB interface 2), maps them to SC2 12-byte gamepad and 45-byte custom reports, and sends them as ATT notifications. Lizard mode is disabled by periodically re-sending a `0x81` ClearDigitalMappings command via `os.write()`. The host's BlueZ hog-ll driver creates `/dev/hidrawN` and `/dev/input/eventN` from the notifications.

See `README.md` for the full architecture diagram. See `src/main_l2cap.py` for the entrypoint.

**Deprecated** — do not use: `deprecated/main.py` (BlueZ GATT, broken), `deprecated/gatt_app.py` (D-Bus GATT objects, not needed).

## Why Raw L2CAP

BlueZ 5.86 on SteamOS has a critical bug: its GATT listener socket is bound to the adapter's **public address**, but BLE connections arrive on the **static random address** (`C2:12:34:56:78:9A`). The kernel's L2CAP layer can't route the ATT channel to the socket. `connect_cb` in `gatt-database.c:646` never fires, no ATT bearer is created, MTU exchange fails, and the connection drops after ~4 seconds.

Debug proof: `bluetoothd -d -n` shows `connected_callback()` fires, but no GATT/ATT/MTU logs appear before `dev_disconnected()` 4 seconds later.

The fix: a raw L2CAP socket on CID 4 bound directly to `C2:12:34:56:78:9A` with `BDADDR_LE_RANDOM`. Confirmed working — MTU exchange and service discovery succeed. See `research/archive/debug-bluetoothd-analysis.md` for the original debug analysis.

## Haptic Architecture

### Three Haptic Paths

1. **Firmware-local haptics** (SC2 nRF52840 only, not available on Deck/Neptune):
   - Trackpad clicks, button feedback, grip touch/detouch generated by firmware's `haptics-sequencer-touchpad` and `haptics-sequencer-gri-v3`
   - Scripts are firmware-internal, host triggers by ID via 0x8F
   - Not available to SpoofDeck — Deck's Neptune lacks the SC2 haptic sequencer

2. **Game rumble via 0x80** (works on any transport):
   - `SDL_RumbleJoystick()` → `SDL_hid_write()` → UHID → hog-ll → ATT Write → `_on_haptic_write()` → Neptune ERM motors
   - NOT gated by `[esi+0x17c]`

3. **Steam-generated haptics via 0x8F** (USB/Dongle ONLY):
   - Steam UI → `CPulseHapticWorkItem` → 0x8F sub-command → firmware haptic script trigger
   - **BLOCKED on BLE by upstream dispatch** — not just the gate at `[esi+0x17c]`
   - A real SC2 only gets these haptics through the Puck dongle (USB, state 1-2)
   - LD_AUDIT patching investigated: scheduler at `0x123e5d0` is never called for BLE controllers. Block is upstream in controller setup/dispatch, before vtable checks.
   - See `docs/findings-backlog.md` for full analysis

**Not working:**
- **Steam-generated haptics** (trackpad clicks, UI feedback via 0x8F) — Architecturally blocked. The haptic scheduler at `0x123e5d0` is never called for BLE controllers (GDB confirmed). `CPulseHapticWorkItem` fires with 0.0ms runtime — the work item completes instantly because it never enters the scheduler. The block happens upstream of the scheduler, before vtable checks. Real SC2 also doesn't get these haptics over BLE. LD_AUDIT investigation completed — see `docs/findings-backlog.md` for full analysis.

**Previously broken, now resolved:** Zombie disconnects and encryption errors were caused by stale BlueZ state. Fixed by clearing bond data + host reboot.

## Deck Hardware Details

### Bluetooth Adapter
- Qualcomm QCA, Bluetooth 5.3, supports central + peripheral roles (critical for advertising)
- 16 advertising instances
- Python 3.13.5, BlueZ 5.86, GLib 2.84.3

### Neptune Controller (HID Input)
- VID=0x28DE, PID=0x1205, 3 hidraw interfaces
- Gamepad hidraw: `/dev/hidraw3` (USB interface 2, identified by `HID_PHYS` containing `input2`)
- NO Report ID in descriptor — raw 64 bytes per report
- Report type 0x09 (`ID_CONTROLLER_DECK_STATE`): sticks, triggers, buttons, trackpads, IMU, force sensors
- `hid-steam` driver only generates evdev events when `gamepad_mode` is true (requires Steam). Without Steam, buttons map to keyboard scancodes (lizard mode). Solution: read directly from hidraw.
- Lizard mode re-enables every ~2 seconds; must re-send `0x81` command periodically via `os.write()`
- `HIDIOCSFEATURE` ioctl returns EINVAL on hidraw — use `os.write()` for output reports
- Reference: [InputPlumber](https://github.com/ShadowBlip/InputPlumber)

## SC2 BLE Protocol

Quick reference — see `docs/sc2-protocol.md` for full details.

| Mode | VID | PID |
|------|-----|-----|
| BLE | 0x28DE | 0x1303 |
| USB wired | 0x28DE | 0x1302 |

GATT services (6): GAP (0x1800), GATT (0x1801), HID (0x1812), Battery (0x180F), Device Info (0x180A), Valve Custom SC2.

Input reports: 12-byte gamepad (standard HID), 45-byte SC2 Custom (Report ID 0x45), 47-byte SC2 Custom (Report ID 0x47).

## Known Gotchas (Top 5)

1. **Stale BlueZ state** — The #1 cause of mysterious failures. Host caches pairing keys/CCCDs per-device. After code changes break a connection: `bluetoothctl remove C2:12:34:56:78:9A`, then `sudo rm -rf /var/lib/bluetooth/<HOST_BT_MAC>/C2:12:34:56:78:9A /var/lib/bluetooth/cache && sudo systemctl restart bluetooth`, then restart Deck service. `btusb` module reset does NOT fix this — stale state is in BlueZ user-space. See gotcha #9 in the full list below.
2. **Never use `ControllerMode=le`** in `main.conf` — causes "Not Supported" error
3. **Static BLE address lost on reboot** — must re-apply `config_bt.py` after every bluetooth restart
4. **Python 3.13 BLE socket** — no tuple syntax support; use `ctypes.bind()` with numeric `SOL_BLUETOOTH=10`
5. **hog-ll strips Report ID from output reports** — `_on_haptic_write()` receives 9-byte payload without the `0x80` prefix (type at [0], left speed at [3], right speed at [6]). `forward_report()` uses ATT Write Request (0x12), NOT Write Command (0x52), because CHR_REPORT has `GATT_CHR_PROP_WRITE`.

### Additional Gotchas

6. `btmgmt power-cycles` kill hogp — start hogp AFTER `config_bt.py`
7. `btmgmt info` output is empty — known SteamOS issue, ignore it
8. `steamos-readonly` must be disabled before modifying `/etc/`
9. KDE pairing dialog — host shows dialog during pairing, user must click "yes"
10. Cumulative BlueZ state corruption — repeated connection failures poison HOG driver state. Clear bond data + restart daemon, or full reboot.
11. In-game rumble works, Steam haptics do not — only games calling `SDL_RumbleJoystick()` produce rumble. See `docs/findings-backlog.md`.
12. **Serial format required**: GET_SERIAL response byte[2] must be 0x01 (success status) and serial[0] must be 'F' (0x46). V_strncmp at 0x10c29b3 checks only the first byte. Failure causes "Controller Serial# invalid" (BLOCKING — zombie disconnect). "Deck Controller PCB Serial# invalid" is cosmetic only.

## Important Rules

1. **One change at a time** — never stack fixes. If Fix A doesn't work, revert it, then try Fix B.
2. **Evidence before conclusion** — every finding must cite specific evidence. Tag with confidence level.
3. **Commit progress** — after each meaningful finding: `git add -A && git commit -m "finding: <description>" && git push`
4. **Stale state is the #1 cause of mysterious failures** — every test cycle: clear bond data, restart BlueZ, reconnect.
5. **Structured protocol logging** — `SPOOFDECK_PROTO_LOG=1` enables JSON logging to stderr for ATT opcodes, notifications, haptics, and SC2 commands. Parse with `scripts/extract_proto_trace.py`.

## Firmware RE Summary

Ghidra analysis of SC2 BLE firmware (`ibex_firmware.bin`, 2027 functions, 73K lines) and Puck dongle firmware (`proteus_firmware.bin`, 790 functions, 36K lines). Nordic nRF52840, Zephyr RTOS. 100 total commands in firmware dispatch (we handle 9). 0x8F gate is entirely in steamclient.so — firmware handles 0x8F correctly. 0x80 is the motor output command (NOT 0x8F). 0xf2 is a minimal ACK (6 bytes, no payload). Puck is a transparent ESB→USB relay. Full decompilations in `spoofdeck-ghidra` repo. See `research/triton-firmware-reference.md` and `research/puck-crossref-reference.md` for full findings. Local Ghidra exports: `~/ghidra-projects/exports/32bit/`. Mirrored exports: `~/spoofdeck-ghidra/steamclient_exports/`.
